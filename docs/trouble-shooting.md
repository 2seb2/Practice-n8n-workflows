## 1. 워크플로우가 자동 실행되지 않는 문제 (Publish 누락)

### 증상

- 각 노드를 개별 테스트 실행하면 모두 정상 동작
- 그러나 설정한 폴링 주기가 지나도 워크플로우가 한 번도 자동 실행되지 않음
- Executions 탭에 자동 실행 기록이 전혀 쌓이지 않음

### 원인

워크플로우를 **Publish(활성화)하지 않은 상태**였다.

n8n에서 수동 테스트 실행(Execute)과 트리거 기반 자동 실행은 별개다. 노드별 테스트가 전부 통과해도, 워크플로우를 활성화하지 않으면 Schedule/RSS Trigger는 동작하지 않는다.

### 해결

워크플로우 우측 상단의 활성화 토글을 켜서 Publish. 이후 Executions 탭에서 자동 실행이 기록되는 것을 확인.

### 배운 점

**"테스트가 통과한다"와 "운영 환경에서 돌아간다"는 다른 문제다.** 개발 단계의 수동 검증만 믿지 말고, 배포(활성화) 후 실제 자동 실행 로그를 반드시 확인해야 한다. CI/CD에서 빌드 성공과 배포 완료를 구분하는 것과 같은 원리.

## 2. 중복 검사가 동작하지 않는 문제 (title vs name)

### 증상

이미 Notion에 저장된 뉴스인데도 계속 중복으로 저장됨. 중복 검사 로직 자체는 문제가 없어 보였음.

### 원인

Code 노드에서 Notion 조회 결과를 다음과 같이 참조했다.

```js
// ❌ 잘못된 참조
const notionTitle = item.json.title;
```

그러나 Notion 노드가 실제로 반환하는 데이터에서 제목 필드는 `title`이 아니라 `name`이었다.

```js
// Notion 반환 데이터 실제 구조
item.json.name // ← 제목은 여기에 있음
```

`item.json.title`은 항상 `undefined`였고, RSS 제목과의 비교가 **항상 불일치**로 판정되어 모든 뉴스가 신규로 처리됐다.

### 해결

노드의 실제 Output 구조를 직접 확인한 뒤 참조 필드를 수정.

```js
// ✅ 수정
const notionTitle = item.json.name;
```

수정 후 중복 저장이 사라졌고, 기존 뉴스는 Gemini 호출 없이 워크플로우가 종료되어 불필요한 API 요청도 함께 차단됐다.

### 배운 점

로직이 맞아 보이는데 결과가 틀리면, **추측하지 말고 실제 반환 데이터 구조를 직접 확인**한다. API 문서의 필드명과 통합 도구(n8n)가 가공해 반환하는 필드명은 다를 수 있다.

## 3. Search Database 노드 사용 착오

### 증상

저장된 뉴스 목록을 검색하기 위해 `Search Database` 노드를 사용했지만, 원하는 결과(데이터베이스 안의 뉴스 페이지들)가 반환되지 않음.

### 원인

`Search Database`는 **데이터베이스 자체**(어떤 DB가 존재하는지)를 검색하는 기능이고, **데이터베이스 안의 Row(Page)를 검색하는 기능이 아니었다.** 노드 이름만 보고 용도를 짐작한 것이 착오의 원인.

### 해결

`Get many database pages` 노드로 데이터베이스 내 전체 Row를 가져온 뒤, Code 노드에서 RSS 제목과 비교하는 중복 검사를 직접 구현.

```text
RSS Feed
    ↓
Get many database pages  (저장된 뉴스 전체 조회)
    ↓
Code (중복 여부 검사)
    ↓
신규 뉴스만 → Gemini → Notion 저장
```

### 배운 점

노드 이름으로 기능을 추측하지 말고, 노드의 입출력 스펙을 먼저 확인한다. "Database를 검색한다"는 말은 대상이 DB 목록인지 DB 내부 데이터인지에 따라 완전히 다른 기능이다.

## 4. Notion Database를 찾지 못하는 문제

### 증상

```text
Could not find database with ID ...
```

또는

```text
No output data returned
```

### 원인

두 가지가 겹쳐 있었다.

1. **인라인(Inline) 데이터베이스**를 사용하고 있었음 — API 접근에 제약이 있음
2. n8n Integration이 해당 데이터베이스에 **연결(공유)되어 있지 않았음** — Notion은 Integration을 워크스페이스에 추가하는 것과 개별 페이지/DB에 연결하는 것이 별개 단계

### 해결

1. 데이터베이스를 **Full Page Database**로 변경
2. 데이터베이스 페이지에서 n8n Integration을 직접 연결
3. `Get many databases`로 정상 조회되는지 확인 후 다음 단계 진행

### 배운 점

Notion API 연동 문제의 대부분은 코드가 아니라 **권한 연결 단계**에서 발생한다. "Credential 등록 완료 = 접근 가능"이 아니며, 대상 리소스별로 Integration 연결이 필요하다.

## 5. Notion 속성 매핑 오류

### 증상

Gemini 요약 결과가 Notion의 Summary 컬럼에 저장되지 않음.

### 원인

`Key Name` 자리에 값(Expression)을 입력했다. 속성 **이름**이 들어가야 할 곳에 속성 **값**을 넣은 것.

```text
❌ 잘못된 매핑
Key Name: {{$node["Message a model"]...}}
```

### 해결

속성 이름과 값을 분리해 매핑.

```text
✅ 올바른 매핑
Key Name    : Summary
Text (Value): {{$node["Message a model"]...}}
```

### 배운 점

Key-Value 매핑 UI에서 오류가 나면 "무엇이 이름이고 무엇이 값인지"부터 다시 확인한다. 에러 메시지 없이 조용히 저장만 안 되는 경우, 매핑 구조 착오일 가능성이 높다.

## 6. Gemini 응답 형식이 일정하지 않은 문제

### 증상

- 응답 앞에 "안녕하세요", "네, 요약해드리겠습니다" 같은 인사말이 붙어 그대로 Notion에 저장됨
- 실행할 때마다 응답 형식이 달라짐
- 헤딩·볼드 등 마크다운이 과하게 섞여, 표(Database) 안에서 읽기 불편함

### 원인

프롬프트에서 **출력 형식을 제한하지 않았다.** 저장 위치가 Notion Database의 표 안이라는 점을 고려하면 마크다운 서식은 오히려 가독성을 해치는데, 이 용도를 프롬프트에 반영하지 않았던 것.

### 해결

용도(표 안에 들어갈 짧은 텍스트)를 기준으로 프롬프트에 출력 규칙을 명시.

- 인사말·부연 설명 없이 결과만 출력
- 형식 고정: 뉴스 요약 2문장 + 업무 자동화 아이디어 3개
- 마크다운 서식 사용 금지

반복 테스트하며 규칙을 다듬은 결과, 매 실행마다 동일한 형태의 결과가 저장되도록 개선.

### 배운 점

프롬프트 설계의 기준은 "AI가 잘 답하는가"가 아니라 **"결과물이 쓰일 곳에서 쓸 만한가"**다. 출력이 파이프라인의 다음 단계(저장, 표시)로 넘어가는 자동화에서는 형식 고정이 필수이며, 형식 제약은 반복 테스트로 검증해야 한다.

## 7. Notion Credential은 정상인데 Database 목록이 조회되지 않는 문제

### 증상

Credential 연결 테스트는 통과하는데, 노드에서 Database 목록이 뜨지 않음.

### 원인

문제 4와 같은 계열. Integration의 토큰 자체는 유효하지만, 조회하려는 Database에 Integration 연결 권한이 누락되어 있었다. 추가로, 연결 직후에는 반영에 **약간의 지연**이 있어 바로 목록에 나타나지 않는 경우도 있었다.

### 해결

Database에 n8n Integration을 연결하고, 잠시 후 다시 조회하여 정상 표시 확인.

### 배운 점

외부 서비스 연동에서 "인증 성공"은 여러 단계 중 첫 단계일 뿐이다. 인증(Authentication)과 개별 리소스에 대한 권한(Authorization)을 구분해서 점검하고, 설정 변경이 즉시 반영되지 않을 수 있다는 점도 감안한다.

## 전체 구성 요약

```text
RSS Feed Trigger (Schedule)
        ↓
Get many database pages   ← Notion 저장 목록 조회
        ↓
Code: 중복 검사            ← RSS 제목 vs Notion name 비교
        ↓
신규 뉴스만 통과
        ↓
Gemini API                ← 형식 고정 프롬프트 (요약 2문장 + 아이디어 3개)
        ↓
Notion: Create Database Page
```

**환경:** n8n (Docker Compose) / Google Gemini API / Notion API / Google News RSS
